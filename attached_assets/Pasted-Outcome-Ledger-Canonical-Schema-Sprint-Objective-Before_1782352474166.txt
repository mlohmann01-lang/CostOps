Outcome Ledger Canonical Schema Sprint

## Objective

Before redesigning the Outcome Ledger UI, define the canonical data model.

The Outcome Ledger is not a savings table.

It is the **financial chain of custody for technology value**.

Each outcome must be treated as a financial asset under governance, carrying value through:

```text
Projected → Approved → Executed → Verified → Finance Confirmed → Protected
```

Do not build another synthetic dashboard until this schema is explicit.

---

## Core Principle

Projected value and realised value must not overwrite each other.

Each stage has its own value position.

An outcome should be able to answer:

```text
What was expected?
What was approved?
What was executed?
What was verified?
What did finance confirm?
What value was protected from drift?
Where did value leak?
What evidence supports each transition?
```

## Transition Rules

Implement transition guards.

### PROJECTED → APPROVED

Requires:

```text
owner
business case / approval reason
approval evidence
approvedValue
```

### APPROVED → EXECUTED

Requires:

```text
governed action id
execution evidence
executedValue
```

### EXECUTED → VERIFIED

Requires:

```text
post-action measurement evidence
verifiedValue
```

### VERIFIED → FINANCE_CONFIRMED

Requires:

```text
finance reconciliation evidence
financeConfirmedValue
```

### FINANCE_CONFIRMED → PROTECTED

Requires:

```text
drift monitor / retention evidence
protectedValue
```

Failed / drifted states must require:

```text
reason
evidence or system event
```

## Leakage Calculation

Add a deterministic helper.

```ts
export function calculateOutcomeLeakage(outcome: OutcomeRecord) {
  const projected = outcome.projectedValue ?? 0;
  const approved = outcome.approvedValue ?? projected;
  const executed = outcome.executedValue ?? approved;
  const verified = outcome.verifiedValue ?? executed;
  const finance = outcome.financeConfirmedValue ?? verified;
  const protectedValue = outcome.protectedValue ?? finance;

  return {
    approvalLeakage: projected - approved,
    executionLeakage: approved - executed,
    verificationLeakage: executed - verified,
    financeLeakage: verified - finance,
    driftLeakage: finance - protectedValue,
    totalValueLeakage: projected - protectedValue,
  };
}
```

Important:

```text
Missing later-stage values should not be treated as realised value in UI.
```

The helper can calculate leakage defaults, but the UI must still distinguish:

```text
not reached
pending
zero value
actual zero
```

## Demo Data Rules

Existing demo data may be adapted.

But it must obey the canonical schema.

Do not create fake lifecycle values that contradict stage state.

Examples:

If lifecycleStage is `PROJECTED`, then only projectedValue should exist.

If lifecycleStage is `VERIFIED`, then projected / approved / executed / verified may exist, but financeConfirmedValue and protectedValue should remain undefined unless actually staged.

If lifecycleStage is `PROTECTED`, then all lifecycle values may exist.
