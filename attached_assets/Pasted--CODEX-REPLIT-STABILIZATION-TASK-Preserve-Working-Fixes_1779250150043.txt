# Correct Fix Strategy

## 1. Tenant Fallback

`requireTenantContext` may fallback to `default` only when:

```ts
NODE_ENV !== "production"
```

or explicit env flag:

```ts
ALLOW_DEFAULT_TENANT=true
```

In production, missing tenant header must still reject.

Add test coverage.

---

## 2. Schema Mismatch

Do not permanently delete advanced schema fields.

Instead:

* identify actual DB columns
* identify schema columns
* create migration plan
* add compatibility adapter if needed
* avoid selecting non-existent columns in live routes until migration is applied

Preferred interim fix:

* route-level explicit column select for current live pages
* keep schema model intact if needed by future modules
* add migration backlog doc

Do not use schema deletion as the default fix.

---

## 3. Route Registry

Restore route registry entries unless they are provably broken.

If routes crash due to schema mismatch, gate them safely with:

* readiness response
* 501 not enabled
* degraded mode
* feature flag
* clear JSON error

Do not remove them silently.

---

## 4. Connectors Route

Restore connector route functionality.

If DB-backed connector tables do not exist yet, return deterministic readiness/degraded responses.

Do not delete connector surface area.

Connector UX depends on this.

---

## 5. App Ghost Pages

It is acceptable to remove broken nav exposure.

But do not delete page imports/routes permanently without replacing them with:

* stable placeholder
* “not enabled yet”
* feature-flag guard
* safe route shell

The app should not crash, but future product surfaces should not disappear from codebase blindly.

---

## 6. Recommendations / Execution Compatibility

Keep simple working endpoints for:

* list recommendations
* get recommendation by id
* approve
* reject
* ledger write/update

But preserve governed execution compatibility and economic operations direction.
